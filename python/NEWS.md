# smooth (Python) NEWS

Release history of the Python implementation of the **smooth** forecasting package.


## v1.0.6 (unreleased)

Bugfixes:
* `msdecompose()` ‚Äî fixed three IEEE-754 summation-order mismatches with R that were producing ULP-level seasonal-init differences. The undamped multiplicative-trend recursion in ETS(M,M,M) amplifies those into completely different NLopt basins on chaotic configurations (e.g. on `taylor` with `lags=[48]` Python returned `logLik = ‚àí28 394` while R reached `‚àí29 096`; on `lags=[48, 336]` Python returned `‚àí73 254` while R reached `‚àí27 634` ‚Äî gap of ~45 600). Three swaps in `core/utils/utils.py`:
    1. The seasonal centring mean (`:402`) and the trend-from-diffs mean (`:418`) now go through a new `_fsum_nanmean` helper backed by `math.fsum` (Shewchuk exact summation), matching R's 80-bit `LDOUBLE`-accumulator `mean()` to ULP.
    2. The inner seasonal smoother ‚Äî R's `stats::filter(weights = rep(1,N)/N)` at `R/msdecompose.R:186` ‚Äî was the dominant gap. R's C-level `cfilter` walks the input from the **last** element down to the first, accumulating `value * (1/N)` in IEEE-double. The default `np.mean` (pairwise sum), `math.fsum` (Shewchuk) and naive forward sum all give different bit patterns on the same input. A new `_r_filter_mean` helper at `:14` replicates R's exact summation order ‚Äî verified bit-for-bit on `taylor` at lag=48 across all 84-element seasonal slices.
  After all three swaps, every msdecompose output (level, trend, seasonal patterns) is byte-identical to R on `taylor` for single lags `[48]`, `[49]`, `[50]`, `[96]` and the multi-seasonal `[48, 336]` case, and `ADAM(model='MMM').fit(taylor).loglik` now matches R to 4 decimal places on all of those (single-lag 48 fits the same logLik `‚àí29096.16` as R; previously diverged by ~700). No public API change.

Changes:
* `CES(seasonality, lags, initial, a, b, loss, h, holdout, bounds, ic, ...)` and `AutoCES(seasonality, lags, initial, ic, loss, h, holdout, bounds, ...)` ‚Äî Python ports of R's `ces()` (R/adam-ces.R) and `auto.ces()` (R/autoces.R). Complex Exponential Smoothing in state-space form, with complex-valued smoothing parameters capturing level and "potential" dynamics. Supports the four R seasonality modes (`"none"` / `"simple"` / `"partial"` / `"full"`) and the full `initial` ladder (`"backcasting"` / `"optimal"` / `"two-stage"` / `"complete"`). The two-stage NLopt path (BOBYQA stage-1 ‚Üí Nelder-Mead stage-2) matches R's strict-parity trajectory only with `nlopt>=2.10.0`; the fit emits a `RuntimeWarning` otherwise. `AutoCES.fit()` prunes the seasonality pool by sample size (mirrors R's autoces lines 88-146) and picks the best by `ic`. Both classes are directly importable from `smooth` (`from smooth import CES, AutoCES`). R-parity tests at `tests/test_ces.py` (opt-in via the `r_parity` marker; reference data generated by `tests/R scripts/ces_reference.R`).
* `msdecompose()` now routes its two OLS calls (global smoother at `core/utils/utils.py:299` and NA imputation at `core/utils/utils.py:344`) through a new shared C++ helper exposed as `smooth.adam_general._ols.ols`. The backend is `src/headers/olsCore.h` (pivoted QR with a scale-invariant LINPACK-dqrls-style rank cutoff) ‚Äî the same header the R build now calls from `R/msdecompose.R`. This closes a ULP-level gap in the seeded initial level / trend on multi-parameter ETS models with `initial="optimal"` / `"two-stage"` (the global smoother case now matches R to `rtol=1e-13`). The change also fixes a previously latent rank-determination issue on the NA-imputation path: a naive `arma::solve` rank cutoff was too aggressive on raw Vandermonde design matrices and dropped the lower-order polynomial columns; the new `|R(i,i)| / ||X.col(P(i))||` criterion keeps them. The new `_ols` module installs alongside `_adamCore` / `_eigenCalc` / `_numDeriv`. R-parity test added at `tests/test_msdecompose_r_parity.py` (opt-in via the `r_parity` marker).
* `OM.simulate(nsim, seed, obs, randomizer, **randomizer_kwargs)` and `OMG.simulate(nsim, seed, obs, **kwargs)` ‚Äî Python ports of R's `simulate.om` / `simulate.omg` (R/om.R:2272-2342, R/omg.R:1945-1997). `OM.simulate` calls `ADAM.simulate` via `super()` to obtain the latent ETS series, applies `om_link_function` (`core/utils/om_cost.py:17`) to map it to a probability, and draws 0/1 occurrence indicators via `rng.binomial(1, prob)`. `OMG.simulate` simulates both sub-models (with deterministic derived seeds `seed` and `seed+1`) and combines their **latent** series via `omg_link_function` (`core/utils/omg_cost.py:23`). `SimulateResult` gains a new `latent` field that exposes the pre-link state-space output so OMG can consume OM's latent series. The default randomizer for OM is a Gaussian sampler keyed on the empirical residual std (substituting for the missing `plogis` branch in `generate_errors`); a callable `randomizer` argument bypasses that and enables the plug-in-numbers R-parity trick. New `tests/test_om_simulate.py` covers shapes / determinism / seasonal lags / xreg / ARMA / OMG composite. New `r_parity`-marked `tests/test_om_simulate_r_parity.py` exercises the plug-in-numbers path against R ‚Äî the OMG combined-probability case matches R to `atol=1e-9`; three companion cases are marked `xfail` with explanatory notes (OM-fit-parity, Python's multiplicative-seasonal numerics, an upstream R-side ARMA-simulate bug). R-side `simulateADAMCore` gained a one-block `randomizer` override (mirrors `sim.es`'s `do.call` pattern) so the same plug-in-numbers trick now works for `simulate.adam` / `simulate.om` / `simulate.omg` on the R side. `ADAM.simulate` itself was tweaked to accept the structured-dict `persistence` form OM stores, falling back to the raw `vec_g` column-vector when the prepared persistence is a dict.
* `sim_ces(seasonality, obs, nsim, frequency, a, b, initial, randomizer, probability, seed, **randomizer_kwargs)` ‚Äî Python port of R's `sim.ces` (R/simces.R:78-471). Generates Complex Exponential Smoothing series for all four seasonality flavours (`"none"` / `"simple"` / `"partial"` / `"full"`). The complex smoothing parameters `a` (and `b` for `"full"`) are drawn from the "banana"-shaped stability region ‚Äî three circle conditions on the real/imaginary plane (R/simces.R:108-110) ‚Äî via reject sampling capped at 1000 tries. The state-space layout (transition matrix in companion form, persistence built from `Re(a) ¬± Im(a)`) matches R/simces.R:295-311 verbatim, and the model label is `"CES(seasonality)"` (or `"iCES(...)"` for `probability < 1`).
* `sim_gum(orders, lags, obs, nsim, frequency, measurement, transition, persistence, initial, randomizer, probability, seed, **randomizer_kwargs)` ‚Äî Python port of R's `sim.gum` (R/simgum.R:76-436). Generates Generalised Unobserved-components series with arbitrary state-space structure. The caller can supply `measurement`, `transition` and `persistence` matrices verbatim, or let any subset be drawn randomly under an eigenvalue-stability rejection sampler that consumes `smooth_eigens` from the existing `_eigenCalc` pybind module ‚Äî the same eigenvalue helper `ADAM.confint` already uses for `bounds="admissible"`. The rejection loop caps at 2000 tries per sim. Flat caller-supplied `transition` and `initial` vectors are reshaped column-major to match R's `matrix(...)` semantics (R/simgum.R:271, 280). Model label is `"GUM(o1[L1],o2[L2],...)"`.
* Both new simulators accept the same callable-randomizer form as `sim_es` / `sim_ssarima` / `sim_oes`, so the plug-in-numbers parity-test trick extends to them ‚Äî `test_sim_ces_r_parity.py` and `test_sim_gum_r_parity.py` confirm `"none"`/`"full"` CES and 1-/2-component GUM match R to `atol=1e-10` (or `1e-9` for the CES path which has extra centring) when the same errors and the same pinned smoothing parameters / matrices feed both sides.
* All six standalone simulators (`sim_es`, `sim_ssarima`, `sim_sma`, `sim_oes`, `sim_ces`, `sim_gum`) plus the `SimulateResult` dataclass are now directly importable from the top-level `smooth` package ‚Äî e.g. `from smooth import sim_es, sim_ces, sim_gum`.
* `sim_oes(model, obs, nsim, frequency, occurrence, bounds, randomizer, persistence, phi, initial, initial_season, model_b, persistence_b, phi_b, initial_b, initial_season_b, seed, **randomizer_kwargs)` ‚Äî Python port of R's `sim.oes` (R/simoes.R:84-152). Internally calls `sim_es` once or twice (depending on the `occurrence` scheme) and combines the two ETS outputs into an occurrence-probability series. Supports R's four occurrence types: `"odds-ratio"` (`prob = a/(1+a)` from model A only), `"inverse-odds-ratio"` (`prob = 1/(1+b)` from model B only), `"direct"` (clip A's data to `[0, 1]`), `"general"` (`prob = 1/(1 + b/a)` from both). The per-error-type transform (`identity` for multiplicative-error ETS, `exp` for additive) matches R/simoes.R:131-133. Returns a `SimulateResult` with `data` holding the probability series, `model_a` / `model_b` carrying the sub-model results, `occurrence_type` set, and `log_lik = sum(log(prob))` matching R/simoes.R:138-143. Sub-models receive deterministic derived seeds (`seed` and `seed+1`) so the joint result is reproducible. The `__str__` mirrors R's `print.oes.sim` (R/methods.R:2149-2174) ‚Äî header line, "Number of observations in each series:", and the true log-likelihood. A new `r_parity`-marked `test_sim_oes_r_parity.py` confirms the probability series matches R to `atol=1e-10` across all four occurrence types via the plug-in-numbers trick (a stateful `ourFunction` doles out sequential slices of a pre-generated error vector to each `sim.es` sub-call).
* `sim_ssarima(orders, lags, obs, nsim, frequency, arma, constant, initial, bounds, randomizer, probability, seed, **randomizer_kwargs)` ‚Äî Python port of R's `sim.ssarima` (R/simssarima.R:83-752). Generates state-space SARIMA series via the shared C++ `adamCore::simulate` kernel. AR / I / MA polynomials are expanded via the existing `adam_polynomialiser` wrapper (`core/utils/polynomials.py`), the ARI tail fills the first column of the transition matrix `F`, and the persistence vector receives `‚àíARI + MA` ‚Äî same companion-form layout R uses. Random AR / MA draws under `bounds="admissible"` reject coefficients whose polynomial roots fall inside the unit circle (matches R's `polyroot` check), with a `max_tries=1000` cap. `constant=False`/`True`/numeric matches R's three-way constant handling, and the model label is auto-built as `"ARIMA(p,d,q)"` / `"SARIMA(p1,d1,q1)[L1](p2,d2,q2)[L2]..."` with `" with constant"` / `" with drift"` suffixes per R/simssarima.R:716-740.
* `sim_sma(order, obs, nsim, frequency, initial, randomizer, probability, seed, **randomizer_kwargs)` ‚Äî Python port of R's `sim.sma` (R/simsma.R:48-91). Thin wrapper that builds `orders={"ar": [order], "i": [0], "ma": [0]}` and `arma={"ar": [1/order]*order}` and delegates to `sim_ssarima(bounds="none")`. Strips the `arma` / `constant` fields from the result and relabels the model as `"SMA(<order>)"` (or `"iSMA(<order>)"` when `probability < 1`), matching R's post-processing block.
* Both new simulators accept the same callable-randomizer form as `sim_es`, so the same plug-in-numbers parity-test trick works ‚Äî `test_sim_ssarima_r_parity.py` drives R's `sim.ssarima` and `sim.sma` via `randomizer="ourFunction"` with pre-generated errors and confirms the generated data agrees with Python to ``atol=1e-10`` (the C++ kernel is byte-identical when the inputs are).
* `sim_es(model, obs, nsim, frequency, persistence, phi, initial, initial_season, bounds, randomizer, probability, seed, **randomizer_kwargs)` ‚Äî Python port of R's `sim.es` (R/simes.R:113-615). Generates ETS series from a model specification by driving the shared C++ `adamCore::simulate` kernel that R uses. The `randomizer` argument accepts either an R-style name (`"rnorm"`, `"rlnorm"`, `"rt"`, `"rlaplace"`, `"rs"`, `"rgnorm"`, `"rgamma"`, `"rinvgauss"`, `"ralaplace"`, `"rbeta"`) **or** a plain Python callable `f(n) -> ndarray` ‚Äî the latter form mirrors R's `do.call(randomizer, ...)` pattern and is what powers the R-Python parity tests (`tests/test_sim_es_r_parity.py`): feed identical pre-generated errors to both sides via `randomizer="ourFunction"` in R and `randomizer=feed` in Python; the same C++ kernel then produces byte-identical output. Returns a `SimulateResult` dataclass with R-style `__repr__` / `__str__` that reproduces `print.smooth.sim` (R/methods.R:1952) line-for-line ‚Äî header line, persistence vector with `alpha`/`beta`/`gamma` row labels, optional `Phi:` and occurrence line, true likelihood.
* `ADAM.simulate(nsim, seed, obs, randomizer, **randomizer_kwargs)` ‚Äî Python port of R's `simulate.adam` (R/adam.R:7365-7611). Re-simulates `obs` observations from a fitted model using the same matrices the optimiser converged on. Default error distribution and scale come from the model's fitted `distribution` / `scale` (R's switch over `dnorm`/`dlaplace`/`ds`/`dgnorm`/`dlogis`/`dt`/`dalaplace`/`dlnorm`/`dinvgauss`/`dgamma`/`dllaplace`/`dls`/`dlgnorm`); a `randomizer` callable can override it. `ES`, `MSARIMA`, `SMA` inherit the method automatically. The result `model` string is suffixed with `" estimated via adam()"` so `print(sim)` reproduces R's `print.adam.sim` (R/adam.R:7613). The companion `SimulateResult` dataclass is re-exported from the top-level `smooth` namespace.
* `python/CLAUDE.md` ‚Äî added an explicit carve-out under "Never add a `frequency` parameter" stating that the standalone `sim_*` simulators (and **only** them) keep R's `frequency=<int>` parameter, because they have no fitted state or input series to infer seasonality from. The rule remains in force for `ADAM`, `ES`, `OM`, etc.
* `_adam_general.adam_simulator` now passes `nComponents` to the underlying `_adamCore.adamCore` constructor (previously omitted ‚Äî the constructor binding requires it, so the wrapper raised `TypeError` whenever a caller drove the C++ simulator directly). Aligns with `adam_reforecast` and the other wrappers in the same file.


## v1.0.5 (unreleased)

Changes:
* `ADAM.predict(interval="complete"|"confidence")` now accepts a `seed` keyword that is forwarded to the underlying `ADAM.reforecast()` call, so the Monte-Carlo parameter / error draws are reproducible across runs and platforms. Other `interval` modes ignore `seed` (they don't go through `reforecast`). Closes a flake in `test_interval_confidence_narrower_than_complete` on macOS-latest where MC noise at the predict-default `nsim` occasionally flipped the confidence-vs-prediction width inequality.
* Windows wheels now link against BLAS + LAPACK (via the vcpkg-installed OpenBLAS + LAPACK that the CI workflow already provisions through `CIBW_BEFORE_ALL_WINDOWS`). `python/CMakeLists.txt` no longer hard-codes `ARMA_DONT_USE_BLAS` / `ARMA_DONT_USE_LAPACK` on `WIN32`; the same `find_package(BLAS)` / `find_package(LAPACK)` flow runs on every platform, with a documented fallback to disabled-LAPACK only when the dependency is genuinely absent. Closes the Windows-only `RuntimeError: eig_gen(): use of LAPACK must be enabled` that previously broke `ADAM.reapply()` / `ADAM.reforecast()` on ARIMA models and on `bounds="admissible"`.
* `ReapplyResult.plot()` ‚Äî Python port of R's `plot.reapply` (R/reapply.R:816). Renders the refitted matrix as a fan-chart with five nested quantile bands (95% / 80% / 60% / 40% / 20%) shaded lightest-on-the-outside to darkest-on-the-inside, with the actuals overlaid as a thin black line and the original point-estimate fitted values as a dashed purple line. Returns a `matplotlib.figure.Figure`; accepts `ax`, `figsize`, `title`, `ylim`, `ylabel`, `legend` kwargs. The other R method on the `reapply` class ‚Äî `print.reapply` ‚Äî was already covered by Python's existing `__repr__` / `__str__` on `ReapplyResult`.
* `ADAM.reapply()` / `ADAM.reforecast()` now cover **ARIMA models**, **`bounds="admissible"` ETS** and **external regressors** (numeric xreg) ‚Äî three branches that previously raised `NotImplementedError`. ARIMA uses the existing `adam_polynomialiser()` wrapper (`core/utils/polynomials.py`) to expand each per-draw `(phi, theta)` slice into the AR / I / ARI / MA polynomials, fills `arr_f` / `mat_g` per `non_zero_ari` / `non_zero_ma`, and propagates initial ARIMA states through the polynomial transformation for `initial="optimal"`/`"two-stage"`. Admissible-bounds clipping reuses `eigen_bounds()` / `ar_polynomial_bounds()` (`core/utils/bounds.py`) ‚Äî the same helpers `confint` already uses. xreg `reforecast` accepts a future-`X` array (or falls back to the last `h` in-sample rows with a warning, matching R). `_component_names_for_states()` now emits `ARIMAState1..N` and xreg-name labels alongside ETS components so `persistence` / state-cube row indices are R-shaped. `bootstrap=True` forwards `nsim=nsim` to `vcov` (R/reapply.R:95), so the bootstrap covariance uses the same replicate count as the MVN draw. `ES` / `MSARIMA` inherit the new behaviour automatically (they're thin `ADAM` subclasses). `CES` / `GUM` remain out of scope until the classes themselves are ported. A new `r_parity`-marked `test_reapply_reforecast_r_parity.py` drives R from pytest and compares distributional summaries (parameter-draw means, forecast trimmed means) within Monte-Carlo tolerance.
* `ADAM.predict(interval="complete")` and `ADAM.predict(interval="confidence")` now dispatch to `ADAM.reforecast()` and return a stock `ForecastResult` ‚Äî previously the two strings were accepted in the `Literal` annotation but raised at the forecaster dispatch. `"complete"` is R's alias for `reforecast(..., interval="prediction")` (joint parameter + prediction uncertainty), and `"confidence"` averages over error draws first then quantiles across parameter draws (parameter uncertainty only). Default `nsim=100` for both modes whenever the caller leaves `predict`'s `nsim` at the predict-default of 10000, matching R's `forecast.adam`.
* `ADAM.reforecast(h, interval, level, side, cumulative, nsim, ...)` ‚Äî Python port of R's `reforecast.adam` (R/reapply.R:941). Internally calls `reapply()` to obtain per-draw state matrices, samples per-distribution errors via `sample_reforecast_errors` (matches R's `reforecast.adam` switch on `dnorm`/`dlaplace`/`ds`/`dgnorm`/`dlogis`/`dt`/`dalaplace`/`dlnorm`/`dinvgauss`/`dgamma`/`dllaplace`/`dls`/`dlgnorm` verbatim) and occurrence draws, runs the shared C++ `adamCore::reforecast` kernel and reduces the resulting `(h, nsim, nsim)` paths cube to a trimmed-mean point forecast plus optional `interval="prediction"` (quantile across all paths) or `interval="confidence"` (mean across error draws first, then quantile across parameter draws). `cumulative=True`, the `h<=0` short-circuit on the refitted matrix, multi-level intervals and the R-style `Lower bound (Q%)` column labels are all supported. Returns a `ReforecastResult` dataclass with `mean`, `lower`, `upper`, `level`, `interval`, `side`, `cumulative`, `h`, `paths` and `model`; a `to_forecast_result()` helper projects onto the stock `ForecastResult` shape. Phase 2 of the port ‚Äî ARIMA and external regressors raise `NotImplementedError` (inherited from `reapply()`).
* `ADAM.reapply(nsim, bootstrap, heuristics, seed, **vcov_kwargs)` ‚Äî Python port of R's `reapply.adam` (R/reapply.R:87). Samples `nsim` parameter vectors from a multivariate normal centred on `coef` with covariance from `vcov()` (PSD-repaired if needed), clips draws to the admissible region (ETS `usual` bounds + xreg `delta`), then re-runs the shared C++ `adamCore::reapply` kernel once per draw. Returns a `ReapplyResult` dataclass with `states (c, n+L, nsim)`, `refitted (n, nsim)`, `transition (c, c, nsim)`, `measurement (n, c, nsim)`, `persistence (c, nsim)`, `profile (c, L, nsim)`, `random_parameters (nsim, k)` plus `time_elapsed`, `y`, `fitted`, `model`, `nsim` ‚Äî exactly the same set of objects R's S3 `reapply` list carries. Phase 1 of the port covers pure-ETS models with `bounds in {"usual", "none"}`; ARIMA, external regressors and `bounds="admissible"` raise `NotImplementedError` for the time being.
* Finite-difference Hessian (shared between R and Python via `src/headers/hessianCore.h`) now uses **per-parameter relative steps** `h·µ¢ = Œµ^(1/4) ¬∑ max(|x·µ¢|, 1)` instead of a single absolute step. The absolute step was catastrophically too small for parameters whose magnitudes matched the data scale ‚Äî `initial="optimal"` / `"two-stage"` ADAM models with initial level / trend / seasonal in `B` produced FI rows dominated by floating-point noise (caught by the `all(FI==0)` retry but still imprecise). With the new rule the relative perturbation is always meaningful: `vcov()` / `confint()` / `summary()` SEs for those parameters now agree with R to better than 2% (previously 5‚Äì15% off). `initial="backcasting"` / `"complete"` ADAM, OM, and OMG are bit-equivalent to before because their parameters all satisfy `|x·µ¢| ‚â§ 1` so `max = 1` and the step is unchanged. R's `vcov.adam` FI-refit path explicitly aliases `initialType="two-stage" ‚Üí "optimal"` (R/adam.R:2728-ish) so the Hessian is computed identically for the two modes; Python doesn't need a parallel change because its FI methods don't branch on `initial_type`. Tests in `test_adam_summary_r_comparison.py` now use `rtol=2e-2` (was `rtol=0.15` on the full vcov, `rtol=0.12` on SEs); a new `aan_two_stage` scenario locks in the parity.
* Full single-step loss menu (`likelihood`, `MSE`, `MAE`, `HAM`, `LASSO`, `RIDGE`) plus custom-callable loss are now wired up for `OM` and `OMG` on both Python and R. Previously Python `OM` only honoured `likelihood`/`MSE`, Python `OMG` was restricted to `likelihood`, and R `om()`/`omg()` accepted the strings in their signatures but the cost dispatch silently routed everything except `likelihood` and `MSE` to MSE. `LASSO`/`RIDGE` reuse ADAM's `(1 ‚àí Œª) ¬∑ ‚àömean(errors¬≤) + Œª ¬∑ penalty(B)` formula (`R/adam.R:894-937`; Python helper `trim_b_for_penalty` factored out of `cost_functions.py` so OM/OMG/ADAM share the exact same B-trimming). The custom callable matches ADAM's `(actual, fitted, B) ‚Üí scalar` API; for OMG the parameter `B` is the joint vector `concat(B_A, B_B)`. Lambda is exposed as `lambda` on the R signature and `reg_lambda` on the Python signature (consistent with ADAM's existing surface).
* `multicov()` method added to `ADAM`, `ES`, `MSARIMA`, `OM`, and `OMG`, matching R's `multicov.adam` / `multicov.smooth`. Returns the `(h, h)` covariance matrix of multi-step-ahead forecast errors as a `pandas.DataFrame` labelled `h1..hh`. Supports `type="analytical"` (closed-form via the existing `covar_anal` / `var_anal` helpers), `type="empirical"` (rolling-origin cross-product `(errors·µÄ errors) / (nobs - h)` using the existing `rmultistep` method ‚Äî both R and Python call the same C++ `adamCore::ferrors` backend, so per-cell residuals are bit-equivalent), and `type="simulated"` (averages over `nsim` simulator paths via the existing `adam_simulator` pybind binding).
* `OM.sigma` / `OM.scale` now return `sqrt(mean(residuals¬≤))` ‚Äî the link-scale residual std-dev ‚Äî instead of `NaN`. Matches R's `oes_old` (R/oes.R:1253) and `oesg_old` (R/oesg.R:1039, 1049): the OM residuals are on the link-transformed (logit / log-odds) scale, so their second moment is a meaningful scale parameter for the underlying ETS, even though there's no equivalent on the probability axis. Makes `OM.multicov(type='analytical')` produce a finite, PSD covariance matrix that agrees with R to ~5% relative tolerance under matched fits. R-side: a new `sigma.om()` S3 method is registered (R/om.R) so `sigma(om_obj)` and `multicov(om_obj)` work on the R side too ‚Äî previously they returned `numeric(0)` / crashed because `sigma.adam`'s distribution switch didn't include `"plogis"`.

Bugfixes:
* Windows wheel (round two): `python/CMakeLists.txt` now resolves BLAS + LAPACK via CONFIG-mode `find_package(OpenBLAS CONFIG)` first, falling back to MODULE-mode `find_package(BLAS)` / `find_package(LAPACK)` only when no CONFIG file is present. The previous `BLA_VENDOR=OpenBLAS` hint wasn't enough ‚Äî MODULE-mode FindBLAS / FindLAPACK still couldn't see vcpkg's install layout, both `BLAS_FOUND` and `LAPACK_FOUND` came back false, the `else()` branch silently compiled `ARMA_DONT_USE_BLAS` / `ARMA_DONT_USE_LAPACK` in, and `eig_gen()` (in `ar_polynomial_bounds` / `eigen_bounds`) still blew up at runtime with `eig_gen(): use of LAPACK must be enabled` on every ARIMA `reapply` / `reforecast` and on `bounds="admissible"`. vcpkg's `openblas` port ships a full `OpenBLASConfig.cmake` and is a complete LAPACK build (BUILD_WITHOUT_LAPACK=OFF), so CONFIG-mode resolves both BLAS and LAPACK in one shot via the `OpenBLAS::OpenBLAS` imported target. A new Windows-only `FATAL_ERROR` guard at the end of the BLAS/LAPACK block now fails the build immediately if LAPACK still isn't found, instead of silently shipping a broken wheel. Linux / macOS are unchanged (their system openblas / brew openblas don't ship a config file and fall through to MODULE-mode).
* `_psd_correct` (`core/adam.py`) ‚Äî used by `ADAM.reapply()` to PSD-repair the vcov matrix before MVN sampling ‚Äî now catches `numpy.linalg.LinAlgError: Eigenvalues did not converge` from `np.linalg.eigvalsh` and falls back to the diagonal-only matrix (PSD by construction). The convergence failure was platform-specific LAPACK iteration noise on Windows wheels, hit by `test_reapply_admissible_runs_and_returns_finite`. Same response as the existing "very-negative-eigenvalue" branch a few lines below.
* `ar_polynomial_bounds` (Python `bounds.py`) now treats `numpy.linalg.LinAlgError: Eigenvalues did not converge` from `np.linalg.eigvals` as "outside the stability region" rather than letting it propagate. The grid search starts at AR coefficient `-5.0` (R's `arPolinomialsBounds` does the same), which produces companion matrices whose eigenvalue iteration occasionally fails to converge on Windows ‚Äî same matrix on Linux converges. Hiding the convergence failure was the wrong response when we had no graceful path, but now that we have one (the search just advances toward the admissible interior, exactly what `_roots_outside` is supposed to signal), it's the right call.
* `ADAM(...)` with the default `distribution=None` no longer prints a spurious `Warning: Unknown distribution: None. Switching to 'default'.` at fit time. `None` is the constructor default and means "auto-select" ‚Äî it now maps to `"default"` silently in `_check_distribution_loss` (`core/checker/parameter_checks.py`). The warning still fires for genuinely-invalid distribution strings.
* `ADAM.sigma` (and the "Error standard deviation:" line in `print(m)` / `m.summary()`) now matches R's `sigma.adam` formula (R/adam.R:4625-4658): `sqrt(SS / df)` where `SS` is a distribution-specific sum of squared residuals and `df = nobs - nparam`. Previously Python returned the optimisation scale parameter (`_prepared["scale"]`) which coincides with the empirical residual std only for the Normal distribution but diverges for `dgnorm` / `dlgnorm` / `dlaplace` / `dgamma` / `dinvgauss`. `ADAM.scale` is now decoupled from `sigma` and exposes the internal optimisation scale directly (matches R's `m$scale` / `extractScale(m)`). For the AirPassengers MAM + dgnorm reference fit, `sigma` now reports `0.0366` matching R exactly (was `0.051`).
* `ADAM(distribution="dgnorm")` (and `"dlgnorm"`) now converges to the same local minimum as R for the same data. Previously Python's NLopt landed in a different basin ‚Äî same cost surface, different optimiser trajectory ‚Äî producing visibly different Œ±/Œ≤/Œ≥/shape and a misleadingly lower loss value. Root cause: Python's `initialiser.py` set the shape parameter's lower bound to `0.25` (a belt-and-braces complement to the soft penalty at `cost_functions.py:429`), while R's `adam_checkOptimizer` uses `1e-10`. NLopt's Nelder-Mead auto-generates its initial simplex from the bound box, so a tighter shape bound changed the simplex even when the boundary wasn't active at the optimum. Loosening Python's shape bound to `1e-10` (`creator/initialiser.py:877`) gives bit-equivalent fits to R for AirPassengers `MAM` + `dgnorm` (and other dgnorm/dlgnorm fits). The soft penalty `if other < 0.25: return 1e10/other` in the cost function remains as the safety net for the genuinely-pathological low-shape region.
* `print(ADAM)` and `ADAM.summary()` now show the estimated `shape` value in the distribution line for `distribution="dgnorm"` (and `"dlgnorm"` when `gnorm_shape` is available), matching R's output. Previously printed `Generalised Normal with shape=?` even after a successful fit. The shape was already estimated and stored on `self.gnorm_shape`; this fix mirrors it onto `self.other = {"shape": ...}` so the printing helper (`_format_distribution` in `utils/printing.py`) can read it.
* `ADAM(initial="optimal")` with a seasonal lag (e.g. `lags=[12]`) no longer crashes with `ValueError: could not broadcast input array from shape (12,) into shape (11,)` during the seasonal-initial filling step. The fix in `core/creator/filler.py` corrects an off-by-one in the slice that copied initial seasonal values out of the optimisation vector `B`: only `lag - 1` independent seasonals are estimated (the `lag`-th value is fixed by the sum-to-zero / product-to-one constraint), so the slice into `B` must also read `lag - 1` elements ‚Äî previously read `lag`, which both clobbered the next parameter and broke broadcasting into the `mat_vt` row. The fix mirrors R's `adam_filler` (R/utils-adam.R near line 970) which uses `B[j + 2:lagsModel[‚Ä¶] - 1]` (i.e. `lag - 1` elements) and increments `j` by `lag - 1`, consistent with the already-correct `j += lag - 1` line in the Python code.
* `ADAM.rmultistep()` (and therefore `multicov(type='empirical')`) no longer crashes with `IndexError: Mat::cols(): indices out of bounds` on models with `i_order >= 1` or seasonal lags greater than 1 ‚Äî i.e. anything where `lags_model_max >= 2` (most ARIMA models, all MSARIMA defaults, ADAM ETS with seasonal lags >= 2). Root cause: `_compute_multistep_errors` pre-sliced `index_lookup_table[:, lags_model_max:]` before passing it to the C++ `adamCore::ferrors`, which then re-applied a `+lagsModelMax` column offset internally ‚Äî a double-offset that walked off the end of the table. The fix matches R's `rmultistep.adam` (R/rmultistep.R:101-111), which passes the full lookup unchanged. The case `lags_model_max == 1` (non-seasonal ETS, ARIMA(p,0,q)) used to work by luck ‚Äî the off-by-one happened to stay in bounds.

* `OM.coef_names` now returns the proper parameter names (`alpha`, `level`, ‚Ä¶) instead of falling back to `b1, b2, ‚Ä¶`. As a result, `OM.confint` correctly clamps lower bounds for persistence parameters (e.g. `alpha >= 0`) ‚Äî previously the fallback names silently disabled the clamping.

* `AutoADAM` now accepts `lags=12` (scalar) in addition to `lags=[12]` (list). The scalar is normalised to a single-element list, matching `ADAM`'s existing behaviour.
* New ARIMA-orders precedence rule shared by `ADAM` and `AutoADAM`: when `orders` (dict) is supplied it takes precedence and the scalar `ar_order` / `i_order` / `ma_order` arguments are ignored (a warning is emitted). When `orders=None` and one of the scalar arguments has a non-zero value, those are used (as fixed orders by default, or as upper bounds for selection when `arima_select=True`). With no ARIMA spec at all, the model is pure ETS ‚Äî `AutoADAM`'s `arima_select` default is now `False` so no ARIMA selection runs unless explicitly requested.
* `AutoADAM(verbose=True)` now prints the distribution-selection progress as it runs (mirrors R's `auto.adam` `"Evaluating models with different distributions... dnorm, dlaplace, ‚Ä¶, Done!"` output) plus the selected distribution and, when applicable, the selected ARIMA orders.
* `OMG.actuals` (Python) and `actuals.omg` (R) now return the binary occurrence indicator with the **same class** as `actuals(om(y))` would on the same series (`ts` / `zoo` / numeric preserved). The R side additionally stores the input series on the top-level `omg` object as `result$data`.
* `actuals()` on an `OMG` sub-model (R: `omg$modelA` / `omg$modelB`, Python: `OMG.model_a` / `OMG.model_b`) now returns the reconstructed *unobservable* value the sub-model was implicitly fitting before the link function: `fitted + residuals` for `Etype="A"`, `fitted * (1 + residuals)` for `Etype="M"`. R dispatches via a new `omg_submodel` S3 class on the sub-models; Python branches on an `_is_omg_submodel` flag set in `OMG._om_from_side`.
* Log-domain scale computations (`dlnorm`, `dllaplace`, `dls`, `dlgnorm`) now route their `log(1 + errors)` (or `log(1 + errors/yFitted)`) through complex arithmetic ‚Äî `log(as.complex(...))` in R, `log((...).astype(complex))` in Python ‚Äî and take the modulus of the resulting complex value. Removes the `RuntimeWarning: invalid value encountered in log` that previously fired when residuals dropped below `-1`, and gives a continuous, finite scale rather than a NaN that propagates downstream.
* `coefbootstrap()` method added to `ADAM`, `ES`, `MSARIMA`, `OM`, and `OMG`, matching R's `coefbootstrap.adam` / `coefbootstrap.om` / `coefbootstrap.omg`. Returns a `BootstrapResult` (the Python analogue of R's `"bootstrap"` S3 class) with the empirical replicate matrix, mean-centred-cross-product covariance, and run metadata. `vcov(bootstrap=True, ‚Ä¶)` and `confint(bootstrap=True, ‚Ä¶)` now dispatch to it (previously raised `NotImplementedError`). `parallel=True` (or `parallel=<int>`) runs replicates concurrently via `joblib.Parallel` ‚Äî optional dependency, install with `pip install "smooth[parallel]"` or `pip install joblib`; if not installed, a one-line warning is emitted and execution falls back to serial. The internal sampler now mirrors R's variable-length contiguous-window resampling (R/adam.R:5011-5024) so the empirical bootstrap distribution agrees with R's to within Monte Carlo noise ‚Äî covered by a new `r_parity`-marked test suite spanning ADAM ETS, ADAM ARIMA, OM, and OMG. Only `method="cr"` (case resampling) is implemented; `method="dsr"` (R's `greybox::dsrboot`) is not yet ported.

Bugfixes:
* `OMG.fitted` (and `model_a.fitted`, `model_b.fitted`) now match R's `omg()` to machine precision on mixed-error scenarios (e.g. `ANA`/`MNM`). The Python OMG post-fit reconstruction was rebuilding the sub-model state-space matrices with the user-requested error/trend/season types (e.g. multiplicative for `MNM`), while R's `omgFinalFit` reuses the forced-additive matrices the joint optimiser saw. The combined link function applied to the rebuilt-multiplicative raw fits drifted from R by up to 5‚Äì10% on the probability scale. Python now mirrors R: shares the joint-optimisation matrices with each sub-model post-fit (`OMG._om_from_side`). Coefficients, log-likelihood and loss were already identical between the two implementations.
* Shared C++ numerical Hessian: `src/headers/hessianCore.h` is now the single source of truth for the finite-difference Hessian used by both R (`vcov.adam` / `vcov.om` / `vcov.omg`, plus `vcov.ces` / `vcov.gum` / `vcov.msarima` for symmetry) and Python (`_numDeriv.hessian`). The previous R-side dependency on `pracma::hessian` is removed (`pracma` dropped from `Imports`). Same algorithm, same step size, same operation order on both sides ‚Äî given the same inputs, the Hessian computation contributes zero divergence between languages. The residual ADAM-with-`initial="optimal"` cross-language gap (~1e-5) is unrelated to the Hessian and traces to NLopt-vs-nloptr converging to non-bit-identical optima on multi-parameter problems with large-magnitude initial states in B; both implementations now share that floor symmetrically.
* `OM.vcov`, `OM.confint`, `OMG.vcov`, `OMG.confint` are now aligned with R's Hessian computation. Three Python-side fixes: (i) the numerical Hessian uses `bounds="none"` (matching R/adam.R:2797 `boundsFI <- "none"`) so finite-difference perturbations near a boundary parameter no longer return the 1e300 penalty and explode the inverse FI; (ii) `invert_fisher_information` takes `abs(diag(...))` to mirror R/adam.R:5226 ("just in case, take absolute values for the diagonal"); (iii) OMG sub-models now receive proper coefficient names (`alpha`, `gamma`, ‚Ä¶) from the joint initialiser instead of falling back to `b1, b2, ‚Ä¶`, restoring the persistence-parameter clamping in `OMG.confint`. Paired with a fix in R's `vcov.om` (R/om.R fitted-model intake) that preserves the original `initialType`/`nIterations` during the FI refit, both OM and OMG vcov / SE / CI now agree with R to machine precision (~1e-9 to 1e-17).
